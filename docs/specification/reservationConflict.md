# 予約競合時の仕様

## 1. 基本方針

複数ユーザーが同時に同じ時間帯を予約する可能性があるため、
予約作成時は **先に予約を確定したユーザーを優先（先勝ち）**とする。

後から予約を実行したユーザーには、予約失敗を通知する。

---

## 2. 予約更新のリアルタイム反映

ユーザーが予約を作成した場合、できるだけ早く他のユーザーの画面にも反映する。

方法としては以下のいずれかを採用する。

|方法|説明|
|---|---|
|ポーリング|一定間隔（例：10〜30秒）で予約状況を再取得|
|WebSocket|予約作成時に全クライアントへ更新通知|

本システムでは実装の簡易性を考慮し、まずは **ポーリング方式**を採用する。

例

```
10秒ごとに予約状況APIを再取得
```

---

## 3. 予約作成処理

ユーザーが「予約する」ボタンを押した場合、

```
予約作成API
```

を呼び出す。

サーバー側では以下を確認する。

```
対象時間帯に既存予約が存在しないか
```

問題なければ予約を作成する。

---

## 4. 予約競合時の処理

予約作成API実行時に
同じ時間帯の予約がすでに存在する場合、

```
予約作成失敗
```

としてレスポンスを返す。

---

## 5. UI表示

予約作成失敗時はダイアログを表示する。

例

```
予約できませんでした

他のユーザーが同じ時間帯の予約を先に行ったため、
この時間帯は予約できなくなりました。

予約状況を更新します。
```

ダイアログ閉じる後

```
予約状況を再取得
```

して画面を更新する。

---

## 6. 処理フロー

```
ユーザーA 予約ボタン押下
ユーザーB 予約ボタン押下

      ↓

サーバー

Aの処理が先に完了
→予約作成成功

Bの処理
→既に予約あり
→予約失敗

      ↓

ユーザーB

エラーダイアログ表示
→予約状況再読み込み
```

---

## 7. UX補足

予約成功時は以下のメッセージを表示する。

```
予約が完了しました
```

その後

```
予約状況画面を更新
```

する。
